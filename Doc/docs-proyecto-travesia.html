Principio de Diseño Central

El sistema usa el DNI o Legajo como identificador universal para cruzar todas las fuentes de datos: notas de parciales cargadas por docentes vía XLS, encuestas completadas en el campus, e información de inscripción académica.

🎯

Separación de Perfiles

El sistema distingue entre alumnos ingresantes (primer año, riesgo de abandono temprano) y alumnos avanzados (años superiores, riesgo de atraso o abandono tardío), aplicando lógicas de procesamiento diferenciadas para cada perfil.

🟢

Verde

Trayectoria regular. Sin alertas activas. Aprobación dentro de parámetros esperados.

🟡

Amarillo

Riesgo moderado. Indicadores por debajo del umbral o encuesta omitida.

🔴

Rojo

Riesgo crítico. Requiere intervención proactiva del sistema de tutoría.

Backend — Módulos Python

🚀

main.py

Punto de entrada de la aplicación FastAPI. Instancia la app, registra todos los routers (ingesta, alumnos, dashboard) y configura el ciclo de vida de la conexión a PostgreSQL a través del ORM SQLModel.

FastAPI Entry Point Router Register Lifespan
🗃

models/models.py

Define todas las entidades del sistema usando SQLModel: Estudiante, Materia, Inscripcion, NotaParcial, Encuesta, PerfilRiesgo, Tutor. Cada clase actúa simultáneamente como tabla SQL y schema Pydantic.

SQLModel Pydantic ORM
📥

routers/ingesta.py

Centraliza los endpoints de carga de información académica. Permite subir archivos XLS con notas de parciales y ausentes (procesados con Pandas/OpenPyXL), e importar respuestas de encuestas provenientes de Moodle.

POST /upload-xls POST /encuesta OpenPyXL
👤

routers/alumnos.py

Endpoints de consulta y gestión de estudiantes. Permite obtener el perfil completo de un alumno (datos personales, historial de notas, estado del semáforo) y listar todos los estudiantes con filtros por carrera, año y nivel de riesgo.

GET /alumnos GET /alumno/{id} CRUD
📊

routers/dashboard.py

Genera y sirve los dashboards interactivos mediante Plotly. Exporta HTML auto-contenido con gráficos del semáforo, distribución de riesgo y evolución académica. Distingue entre vista docente (agregados) y vista alumno (individual).

GET /dashboard/docente GET /dashboard/alumno/{id} Plotly HTML
🧮

services/procesador.py

Núcleo analítico del sistema. Usa Pandas para calcular el porcentaje de aprobación de cada alumno, cruzar datos cuantitativos (notas) con cualitativos (encuestas), y generar los insumos para el motor de reglas. Aplica lógicas diferenciadas por año de cursada.

Pandas Métricas KPIs Cross-data
🚦

services/semaforo.py

Motor de reglas que clasifica a cada estudiante en Verde / Amarillo / Rojo basándose en el porcentaje de aprobación calculado por el procesador y el estado de completitud de encuestas obligatorias. Es el componente central para la toma de decisiones de tutoría.

Rule Engine Clasificación Umbrales
🔔

services/alertas.py

Gestiona el disparo de alertas cuando se detectan situaciones críticas: encuestas obligatorias no completadas, ausencias repetidas, o cambio de semáforo a Rojo. Genera las notificaciones que el sistema de tutoría debe atender con prioridad.

Alertas Triggers Notificaciones

Frontend — Componentes JavaScript

🖥

src/front-end/controllers/iframe_handler.js

Controlador JavaScript que gestiona la inyección dinámica de los dashboards HTML generados por Plotly dentro de los iframes del campus Moodle. Detecta el rol del usuario (docente/alumno) y carga el dashboard correspondiente desde los endpoints de la API. Permite embeber contenido personalizado sin modificar el código base de Moodle.

ES Module iFrame Injection Role Detection Dynamic Load
// Lógica conceptual del módulo const container = document.getElementById('iframe_container'); const userRole = getMoodleUserRole(); // 'docente' | 'alumno' const endpoint = userRole === 'docente' ? '/api/dashboard/docente' : `/api/dashboard/alumno/${userId}`; const iframe = document.createElement('iframe'); iframe.src = endpoint; container.appendChild(iframe);

Flujo de Datos

El sistema orquesta tres fuentes de datos distintas y las transforma en dashboards accionables embebidos en el campus virtual.

Pipeline Completo

1

Ingesta de Datos Cuantitativos (XLS)

Los docentes exportan el listado de notas de parciales y ausentes desde sus sistemas y cargan el archivo XLS a la API mediante el endpoint /upload-xls. El módulo procesador.py con OpenPyXL lee las columnas y normaliza los datos usando el DNI/Legajo como clave de cruce.

Notas parcialesAusenciasInscriptos
2

Ingesta de Datos Cualitativos (Encuestas Moodle)

Las encuestas diseñadas en Moodle (cuestionarios nativos) son respondidas por los alumnos en el campus. Las respuestas son importadas a la API por el endpoint /encuesta, vinculando cada respuesta con el perfil del estudiante vía DNI/Legajo.

AutopercepciónFactores socioeconómicosMotivación
3

Procesamiento y Cruce de Fuentes

El módulo procesador.py cruza ambas fuentes de datos usando Pandas para construir el perfil de riesgo actualizado de cada estudiante. Aplica lógica diferenciada según el año de cursada: métricas de ingresantes vs. métricas de alumnos avanzados.

% AprobaciónEncuesta completadaTendencia
4

Motor de Semáforos y Alertas

semaforo.py recibe el perfil procesado y aplica el conjunto de reglas para clasificar al estudiante en Verde, Amarillo o Rojo. alertas.py dispara notificaciones cuando se detectan condiciones críticas (ej. encuesta no completada).

Rule EngineAlertasSemáforo
5

Generación de Dashboards HTML (Plotly)

El router dashboard.py usa Plotly para generar visualizaciones interactivas del estado del semáforo, distribución de riesgo por cohorte, y evolución académica individual. El output es un archivo HTML auto-contenido.

PlotlyHTML ExportCharts
6

Integración en Moodle vía iFrame

iframe_handler.js detecta el rol del usuario en Moodle e inyecta el dashboard correspondiente como iframe en el campus virtual. El docente ve el dashboard agregado de toda la cohorte; el alumno ve únicamente su propio semáforo y estado académico.

iFrameMoodleDashboard DocenteDashboard Alumno

Notas de Diseño

⚡

PERFIL_RIESGO como snapshot

La tabla PERFIL_RIESGO almacena un snapshot del semáforo en cada cálculo, no solo el estado actual. Esto permite auditoría histórica y análisis de evolución del riesgo a lo largo del tiempo.

🔗

JSONB para respuestas de encuesta

Las respuestas individuales de cada pregunta se almacenan como JSONB en PostgreSQL, lo que permite flexibilidad en el diseño de los cuestionarios sin requerir migraciones de esquema.

🔑

DNI/Legajo como clave natural

El campo dni/legajo tiene restricción UNIQUE sobre la tabla ESTUDIANTE y actúa como clave de cruce entre las importaciones XLS y el perfil en la base de datos.

📋

INTERVENCION_TUTORIA como log

Cada acción de tutoría (llamado, reunión, email) se registra con resultado, creando un historial que permite evaluar la efectividad de las intervenciones sobre el semáforo del estudiante.
