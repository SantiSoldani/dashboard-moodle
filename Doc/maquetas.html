🚦 Diseño de Arquitectura: Semáforos en Moodle

Reporte de evaluación técnica para Aula Puente. Escala: 100-500 alumnos.

Parámetros Base del Proyecto:
  • Cálculo: Notas + Asistencia + Actividades (Ciclo actual, sin historial).
  • Roles: Administrador (gestiona parámetros, aprueba avisos), Profesores (reciben alertas y contactan alumnos), Alumnos (ven su estado propio).
  • Notificaciones: Generadas con Claude API, flujo semi-automático, entregadas vía Moodle Internal Messages.
  • Equipo: Mixto (técnicos y no técnicos).

Opción 1: Google Sheets + Looker Studio + GitHub Pages

La alternativa de menor barrera de entrada técnica. Utiliza el ecosistema gratuito de Google para la persistencia y visualización, integrando la API de Claude mediante una interfaz web estática.

Capas de Procesamiento

  • Ingesta (Capa de Datos): Google Forms (2° año) y subida manual de CSV a Google Sheets (1° año).
  • Lógica y Cálculo: Fórmulas dentro de Google Sheets combinando notas, asistencia y entregas.
  • Presentación (Moodle iFrames): Looker Studio para el Dashboard de Admin. GitHub Pages (HTML/JS) para el perfil del alumno y la interfaz de aprobación de notificaciones.
  • Notificaciones: Interfaz en GitHub Pages llama a la API de Claude para redactar, el Admin aprueba, y mediante JavaScript se hace un POST a la API REST de Moodle (función core_message_send_instant_messages).

Recursos y Tiempos

Tiempo estimado: 2 a 3 semanas

Equipo necesario: Perfil semi-técnico (manejo avanzado de Excel/Sheets, JS básico, conocimiento de Moodle API).

Tecnologías:

Ventajas

  • Infraestructura 100% gratuita y sin mantenimiento de servidores.
  • Looker Studio permite crear dashboards muy visuales sin programar.

Desventajas

  • La lógica en hojas de cálculo puede volverse frágil si se modifican columnas accidentalmente.
  • La seguridad se basa en ocultar URLs (seguridad por oscuridad en GitHub Pages si no se hace un backend).

Opción 2: Supabase + Next.js (Vercel) + Claude API

La opción de grado profesional. Crea una aplicación web completa que actúa como "cerebro" detrás de Moodle, con base de datos real y alta escalabilidad.

Capas de Procesamiento

  • Ingesta (Capa de Datos): Interfaz Next.js para carga de CSV. Formularios generados en Next.js embebidos en Moodle. Base de datos PostgreSQL (Supabase).
  • Lógica y Cálculo: Funciones del lado del servidor en Next.js (Node.js) procesan los pesos de notas, asistencia y entregas.
  • Presentación (Moodle iFrames): Rutas de Next.js protegidas. /admin/dashboard embebido para administradores; /alumno/[id] embebido para estudiantes con validación JWT o Moodle Token.
  • Notificaciones: Panel de control en la App donde el admin ve alumnos críticos, hace un clic, el backend llama a Claude, muestra el borrador, el admin lo edita/aprueba y el backend lo envía por Moodle API.

Recursos y Tiempos

Tiempo estimado: 4 a 6 semanas

Equipo necesario: Desarrollador Web Full Stack (React/Next.js, Node, SQL básico).

Tecnologías:

  • Robusto, seguro y libre de errores accidentales (como borrar una celda de Excel).
  • Interfaz de usuario unificada y diseñada a medida.
  • Los parámetros del semáforo se cambian desde un panel UI muy intuitivo.

Desventajas

  • Requiere conocimientos de programación avanzados para el equipo técnico.
  • Mayor tiempo de desarrollo inicial.

Opción 3: n8n + Airtable + Streamlit

El punto dulce para equipos mixtos. Usa automatización visual (n8n) y bases de datos amigables (Airtable) para gestionar la lógica, delegando el dashboard a Python puro (Streamlit).

Capas de Procesamiento

  • Ingesta (Capa de Datos): Admin sube CSV a Airtable o n8n captura webhooks de encuestas de Moodle y los inserta en Airtable.
  • Lógica y Cálculo: Fórmulas nativas de Airtable (muy parecidas a Excel pero relacionales) calculan el estado (Verde/Amarillo/Rojo).
  • Presentación (Moodle iFrames): Vistas de Airtable embebidas para edición rápida de parámetros. Streamlit Cloud embebido para el Dashboard analítico general.
  • Notificaciones: Una vista en Airtable filtra "Alumnos Críticos". El admin cambia una celda a "Aprobar Aviso". n8n detecta el cambio, llama a Claude API, y luego n8n usa el nodo HTTP para pegarle a la API de Moodle y enviar el mensaje.

Recursos y Tiempos

Tiempo estimado: 3 a 4 semanas

Equipo necesario: Perfil técnico en Python (para Streamlit) y alguien con pensamiento lógico para armar los flujos visuales en n8n.

Tecnologías:

Ventajas

  • Airtable es excelente para equipos no técnicos; pueden ajustar los parámetros visualmente.
  • El flujo semi-automático se resuelve brillante y visualmente con n8n.
  • Streamlit permite dashboards analíticos potentes con muy pocas líneas de Python.

Desventajas

  • Airtable tiene un límite de 1000 registros en la capa gratuita (Para 500 alumnos y 1 solo ciclo actual está perfecto, pero no sobra mucho margen).
  • n8n Cloud tiene límite de ejecuciones gratuitas (se recomienda hostear n8n localmente o en un VPS barato).
